# terraform-provider-leifwind

[![License: MPL-2.0](https://img.shields.io/badge/License-MPL--2.0-brightgreen.svg)](LICENSE)

A Terraform/OpenTofu provider for the **leifwind metadata API** — projects,
entities, and fields that make up leifwind's schema-per-project metadata
model. It manages the control plane only: creating/reading/updating/deleting
metadata objects against a running leifwind backend.

Registry source address: **`leifwind-io/leifwind`**.

This repository also ships a standalone, importable Go client
(`./client`) with zero `terraform-plugin-*` dependencies — see
[`client/README.md`](client/README.md) if you want to talk to the metadata
API from your own Go code without Terraform.

## Install

```hcl
terraform {
  required_providers {
    leifwind = {
      source  = "leifwind-io/leifwind"
      version = ">= 0.1.0" # pin to the latest released version
    }
  }
}
```

The provider is distributed via GitHub Releases and resolvable from both the
[Terraform Registry](https://registry.terraform.io) and the
[OpenTofu Registry](https://registry.opentofu.org) — the same
`leifwind-io/leifwind` source address works with `terraform init` and
`tofu init` alike; no separate registry configuration is needed.

See [`docs/index.md`](docs/index.md) and [`docs/resources`](docs/resources)
/ [`docs/data-sources`](docs/data-sources) for the full provider schema and
runnable examples (generated by `tfplugindocs` — see Development below).

## Authentication

The provider supports two auth paths, mutually exclusive:

| Path | Attributes | Env var fallback | Use case |
| --- | --- | --- | --- |
| Static/delegated token | `token` | `LEIFWIND_TOKEN` | A token acquired elsewhere is handed to the provider as-is. This is the path used by leifwind's delegated apply-runner (LW-44): the frontend authenticates a human user via ZITADEL, and the runner forwards that user's token unchanged, so Terraform operations are attributed to the delegating user, not a service identity. |
| `client_credentials` (M2M) | `issuer`, `client_id`, `client_secret`, `audience` | `LEIFWIND_OIDC_ISSUER`, `LEIFWIND_CLIENT_ID`, `LEIFWIND_CLIENT_SECRET`, `LEIFWIND_OIDC_AUDIENCE` | Operator/local or service-to-service use: the provider exchanges client credentials for a bearer token itself and refreshes it automatically before expiry. |

Both paths also take `endpoint` / `LEIFWIND_ENDPOINT` (the backend base URL).

```hcl
# Delegated/static token (runner path):
provider "leifwind" {
  endpoint = "https://api.leifwind.example"
  token    = var.leifwind_token # or LEIFWIND_TOKEN
}

# Alternative — client_credentials (operator/M2M path):
# provider "leifwind" {
#   endpoint      = "https://api.leifwind.example"
#   issuer        = "https://auth.leifwind.example"   # LEIFWIND_OIDC_ISSUER
#   client_id     = "…"                               # LEIFWIND_CLIENT_ID
#   client_secret = var.client_secret                 # LEIFWIND_CLIENT_SECRET
#   audience      = "326102453042806786"              # LEIFWIND_OIDC_AUDIENCE
# }
```

Complete attribute reference: [`docs/index.md`](docs/index.md).

### Known backend quirks to be aware of

- **Project names are globally unique across all tenants**, not just within
  your organization (LW-71). This is intended — a consequence of the
  schema-per-project design, where each project maps to its own Postgres
  schema and schema names are database-global. Namespace project names per
  organization to avoid collisions. See
  [`docs/resources/project.md`](docs/resources/project.md).
- **Deleting the last field of an entity currently 500s** in the backend's
  `sync_entity_schema` (LW-70). Until the backend fix ships, keep at least
  one field un-managed by your configuration on each entity, or destroy the
  owning `leifwind_entity` instead of deleting every one of its fields
  individually. See [`docs/resources/field.md`](docs/resources/field.md).

## Development

This is a two-module Go workspace:

- `.` — the provider itself, built on `terraform-plugin-framework`.
- `./client` — a standalone Go client for the metadata API (no
  `terraform-plugin-*` dependencies), tagged and released independently as
  `client/vX.Y.Z`.

A committed `go.work` ties the two together for local development and CI so
that the provider always builds against the client's working-tree copy:

```
go 1.25.8

use (
	.
	./client
)
```

Release builds run with `GOWORK=off` so the provider's `go.mod` pin against a
**released** `client/vX.Y.Z` tag is what actually gets built and shipped
(see Release, below).

### Local prerequisites

- **Docker** — the client and acceptance test suites boot a real stack
  (ZITADEL + the leifwind backend + PostgreSQL) via testcontainers.
- **One-time registry login** for the private backend test image:
  `docker login registry.gitlab.com` using a personal access token with the
  `read_registry` scope.
- **Go ≥ 1.25**.
- **`tofu`** (OpenTofu CLI) on `PATH` — the acceptance suite drives real
  `plan`/`apply`/`destroy` cycles through it.

### Common tasks

```bash
make lint     # golangci-lint, both modules
make test     # client module's full test suite (testcontainers, -p 1)
make testacc  # provider acceptance suite (TF_ACC=1, against tofu)
make docs     # regenerate docs/ + examples/ via tfplugindocs
make tidy     # go mod tidy, both modules
```

### GitLab primary, GitHub mirror

Development, code review, and CI all live on GitLab
(`gitlab.com/leifwind/stream/terraform-provider-leifwind`) alongside the rest
of the leifwind backend. A push mirror keeps
`github.com/leifwind-io/terraform-provider-leifwind` in sync, because both
the Terraform Registry and the OpenTofu Registry resolve providers from
public GitHub repositories and GitHub Releases. The GitHub mirror runs no CI
of its own — it exists purely for distribution. The release job explicitly
pushes the release tag to GitHub before invoking `goreleaser`, to avoid
racing the mirror's sync lag against registry ingestion.

## Release

Releases are cut by pushing tags, in this order:

1. `client/vX.Y.Z` — tags the client module. CI runs lint + test only; module
   tags need no build artifacts (the mirror syncs the tag).
2. `vX.Y.Z` — tags the provider. On a protected `v*` tag, GitLab CI:
   - imports the release GPG key,
   - pushes the tag to the GitHub mirror,
   - runs `GOWORK=off goreleaser release --clean`, building and signing
     cross-platform archives and publishing a GitHub Release (which both
     registries index).

Bump the provider's `go.mod` client dependency to the new `client/vX.Y.Z`
before tagging the provider, so the `GOWORK=off` build resolves the release
you just cut rather than an older one.

First-release/registry-onboarding steps (Terraform/OpenTofu Registry
onboarding, flipping the client dependency off its dev-time placeholder pin,
and proving the `GOWORK=off` release build end-to-end) are one-time setup
tracked separately in **LW-68** — not part of this repository's day-to-day
release flow.
